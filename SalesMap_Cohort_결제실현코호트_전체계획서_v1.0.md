## 1. 목적

유입 코호트 대시보드가 `유입 → 리드 전환 → 딜 성사 → 딜 완료`를 **유입월에 고정된 같은 고객 집합**으로 끝까지 추적하도록 만든다. 동시에 `딜 성사 → 딜 완료` 사이의 **결제유예기간(1~6개월)**을 노이즈가 아닌 측정 가능한 신호로 전환해, "언제 얼마가 결제 실현되는지"를 보여주는 현금흐름 지표로 활용한다.

본 계획은 다음 세 가지 우선순위를 단계적으로 구현한다.

1. **유입일 정상화** (최우선) — 정규화 필드 도입, 이관 데이터 분리
2. **anchor-set 기반 재계산** — 유입월에 고객 집합 고정, 이후 전 기간 이벤트 추적
3. **Row1에 `딜 완료`·완료율 추가** — 결제유예 코호트 모델(M+n·3상태·예측·지연) 반영

## 2. 핵심 문제 정의

### 2.1 생성일 ≠ 실제 유입일

2026년 2월 노션 데이터를 일괄 업로드하면서 세일즈맵 `생성일`이 실제 유입일과 어긋났다. `생성일`을 유입일로 간주하면 기존 이관 고객이 2월 신규 유입처럼 집계되어, 딜 전환율 100% 초과·유입 없는 딜 존재·완료 수가 유입 수보다 큰 왜곡이 발생한다.

### 2.2 성사 ≠ 완료 (결제유예 시차)

`딜 성사`(자금집행 완료) 이후 `딜 완료`(결제 완료)까지 1~6개월의 결제유예기간이 존재한다. 과거 이 시차 때문에 최근 성사 딜이 "완료 0건"으로 잡혀 지표가 망가진 것처럼 보였고, 결국 `딜 완료`를 Row1에서 제거했다. 그러나 이는 표시 방식의 문제이며, 코호트 성숙도(M+n)로 모델링하면 해결된다.

### 2.3 기준일 혼합

현재 `computeCohortKPI()`는 `dealMatchesPeriod(deal, range)`로 기간 필터된 데이터에 의존한다. 이는 "유입월에 고정한 anchor set을 이후 월까지 추적"하는 순수 코호트 모델과 다르다. 코호트 기준(유입월)과 성과 발생월 기준이 섞이면 전환율 해석이 왜곡된다.

## 3. 우선순위 1 — 유입일 정상화

### 3.1 정규화 필드

각 고객/조직(또는 딜) 레코드에 다음 필드를 추가한다.

| 필드 | 의미 | 비고 |
| --- | --- | --- |
| `actual_inflow_date` | 실제 최초 유입일 | 코호트 귀속의 기준 |
| `salesmap_created_at` | 세일즈맵 생성일 | 운영/이관 로그용 |
| `migration_flag` | 노션 이관 여부 (true/false) | true면 2월 신규에서 제외 |
| `migration_source` | 원본 출처 | 노션 DB/배치 ID 등 |
| `inflow_confidence` | 실제 유입일 신뢰도 (high/mid/low) | low는 미확인 그룹 |
| `cohort_month` | 코호트 귀속월 | `actual_inflow_date`에서 도출 |
| `event_month_deal_won` | 딜 성사 발생월 | 성과 집계용 |
| `event_month_deal_completed` | 딜 완료 발생월 | 성과 집계용 |

### 3.2 유입일 결정 우선순위

`actual_inflow_date`는 다음 순서로 결정한다.

1. 노션 원본의 최초 유입일
2. 세일즈맵 명시적 유입일 또는 상담 시작일
3. 최초 활동/메모/상담 기록일
4. 최초 리드 생성일
5. 세일즈맵 생성일 — **단, 이관 데이터가 아닌 신규 생성 데이터일 때만**

### 3.3 이관 데이터 분리

| 분류 | 기준 | 처리 |
| --- | --- | --- |
| 신규 유입 | 실제 유입일 2026-02 이후 & 이관 흔적 없음 | 코호트 포함 |
| 기존 이관 | 노션 원본/메모상 과거 유입 확인 | 실제 유입월로 재귀속, 2월 신규 제외 |
| 유입일 미확인 | 실제 유입일 복구 불가 | 별도 그룹 분리, 코호트 계산 제외 |

이관 배치 탐지 조건: 생성일이 2026-02에 집중 / 생성일은 같으나 과거 활동 이력 존재 / 노션 원본에 2월 이전 기록 / 생성 직후 이미 중간 단계 이상 상태.

## 4. 우선순위 2 — anchor-set 기반 재계산

### 4.1 원칙

1. **anchor set 확정**: `cohort_month`별로 고객 집합을 먼저 고정한다.
2. **이벤트 추적**: 그 집합 안에서만 리드 전환·딜 성사·딜 완료 이벤트를 전 기간에 걸쳐 찾는다. 기간 필터(`dealMatchesPeriod`)로 딜을 거르지 않는다.
3. **이중 기준 분리**: 코호트 귀속은 `cohort_month`, 월별 성과는 `event_month_*`로 분리 집계한다.

### 4.2 예시

- 5월 유입 → 5월 딜 성사 → 8월 딜 완료
- 코호트 귀속: **5월** / 코호트 표시: **5월 코호트 M+3 완료** / 월별 성과: **8월 딜 완료 1건**

### 4.3 필터 모드

| 모드 | 기준 |
| --- | --- |
| 코호트 기준 | 실제 유입월 (`cohort_month`) |
| 성과 발생월 기준 | 딜 성사/완료 이벤트 발생월 (`event_month_*`) |

두 기준은 화면에서 명확히 분리 표시하고 혼합하지 않는다.

---

## 5. 우선순위 3 — 결제 실현 코호트 모델 (Row1 `딜 완료`·완료율)

### 5.1 결제 실현 3상태 분할

`딜 성사` 이후 딜을 단순 완료/미완료가 아니라 세 상태로 나눈다.

| 상태 | 기준 | 의미 |
| --- | --- | --- |
| 완료 | 결제 완료 단계 도달 | 매출 실현 |
| 유예 중 (미도래) | 성사 & `성사일 + 6개월` 미경과 | 정상 대기, 실패 아님 |
| 지연 (초과) | 유예 한도 초과 & 미결제 | 회수 관리 대상 |

이 분리가 핵심이다. 셋을 합치면 정상 대기 건이 미완료처럼 보여 지표가 무너진다.

### 5.2 M+n 누적 완료율

`딜 완료`는 절대월이 아니라 **성사 후 경과 개월(M+0~M+6)별 누적 완료율**로 측정한다. 최근 코호트의 빈 셀은 실패가 아니라 "미도래(유예 중)"로 표시한다.

### 5.3 성숙 코호트 기준선 + 예측

- **기준선**: 성사 후 6개월 이상 경과한 성숙 코호트만 모아 회사의 실제 결제 실현 곡선을 산출한다.
- **예측**: 기준 곡선을 최근(미성숙) 코호트에 적용해 향후 결제 실현 건수·금액을 월별로 예측한다 → 현금흐름 예측 산출물.

### 5.4 평균 완료기간

성사일 → 완료일 실제 소요일의 P50/P90을 산출해 유예 한도·지연 기준의 근거로 삼는다.

### 5.5 지연 노후화(aging) 리스트

`성사일 + 6개월 초과 & 미결제` 딜을 자동 추출해 회수 우선순위 리스트로 제공하고 주간 갱신한다.

### 5.6 Row1 재편

```
총 유입 → 리드 전환 → 딜 성사 → 딜 완료(실현) → 완료율(성숙 코호트 기준)
```

별도 영역: 결제 실현 3상태 카드, M+n 히트맵, 완료곡선·예측, 지연 회수 리스트.
`완료율`은 라벨에 *성숙 코호트 기준*임을 명시해 최근 코호트의 낮은 수치와 섞이지 않게 한다.

- 딜 전환율 100% 초과 유입경로

- 리드 전환 없이 성사/완료

## 10. 완료 기준 (Definition of Done)

- 2026-02 생성일을 실제 유입일로 간주하지 않는다.
- 기존 노션 이관 데이터가 신규 유입에서 분리된다.
- 코호트는 `actual_inflow_date` 기준으로 귀속된다.
- 월별 성과는 이벤트 발생월 기준으로 집계된다.
- 5월 유입 → 8월 완료 고객이 5월 코호트 M+3 완료로 누적된다.
- `딜 완료`·완료율이 Row1에 표시되고, 유예 중/지연이 분리된다.
- 미성숙 코호트의 낮은 완료율이 "정상 대기"로 표시되어 지표를 왜곡하지 않는다.
- 독립 재계산과 화면 수치의 차이가 기준일 정의 오류·이관 미분리·분모/분자 불일치 중 하나로 분류·해소된다.
