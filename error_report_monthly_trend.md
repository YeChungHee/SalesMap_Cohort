## 2. 코드 분석 — 데이터 수집 로직

### 2-1. 딜 필터링 기준 (문제의 핵심)

`computeMonthlyPurchaseTypeTrend()` 함수(index.html ~2787행)에서 "완료 딜" 추출 시 다음 3단계 폴백을 사용:

```javascript
const primaryTs = dwells["자금집행 완료"]?.enteredTs || null;   // ① 1순위
const fb1Raw    = d["거래일"];                                  // ② 2순위 폴백
const fb1       = fb1Raw ? new Date(fb1Raw).getTime() : null;
const fb2       = dwells["결제 완료"]?.enteredTs || null;       // ③ 3순위 폴백
const enteredTs = primaryTs || fb1 || fb2 || null;
```

이후 `.filter(d => d.enteredTs)` — **enteredTs가 존재하면 무조건 포함.**

### 2-2. 오류 원인 A — "거래일" 폴백으로 인한 미완료 딜 포함 (주요 원인)

**"거래일"은 실제 자금집행이 완료된 날짜가 아니라, 상담/제안 날짜 또는 예상 거래일로 입력되는 경우가 많음.**

- 세일즈맵에서 "자금집행 완료" 단계에 진입하지 않았더라도 "거래일" 필드에 날짜가 입력된 딜은 모두 집계에 포함됨
- 이로 인해 실제로 진행이 완료되지 않은 업체(진행 중단 / 협의 불발)도 해당 월의 구매 완료 딜로 카운트됨

**해당 오류로 포함된 것으로 추정되는 업체:** 총 14건

### 2-3. 오류 원인 B — "구매이력(purchaseType)" 미입력 시 자동 신규 처리 (부차적 원인)

```javascript
} else {
  cat = "new_";  // 구매이력 필드 미입력 시 신규로 처리
}
```

`d["구매이력"]` 필드가 null 또는 미입력이면 자동으로 "신규"로 분류됨.  
세일즈맵에서 딜 생성 시 "구매이력" 필드를 수동으로 입력하지 않으면 발생.

### 2-4. 오류 원인 C — "해남참농가" 리텐션→신규 오분류

`purchaseType === "Repeat"` 이지만 이전 딜의 `enteredTs`가 없는 경우:

```javascript
const prevDeals = wonDeals.filter(x => x.orgId === d.orgId && x.enteredTs < d.enteredTs);
if (!prevDeals.length) {
  cat = "new_";  // 이전 이력이 없으면 신규 처리
}
```

- "해남참농가"의 이전 딜에 "자금집행 완료", "거래일", "결제 완료" 날짜가 모두 없으면 `prevDeals`가 빈 배열 → 신규 분류
- 또는 이전 딜이 대시보드 집계 기간(최근 12개월) 밖에 있어 `wonDeals` 배열에서 제외된 경우에도 동일 문제 발생
- 추가 가능성: "구매이력" 필드가 null → 원인 B와 복합 작용

## 3. 오류 분류 요약

| 오류 유형 | 원인 코드 | 영향 건수 | 심각도 |
|---|---|---|---|
| 진행하지 않은 딜이 완료로 집계 | "거래일" 폴백 무분별 사용 | 14건 | **높음** |
| 구매이력 미입력 → 신규 자동 분류 | `purchaseType === null` 처리 | 미확인 | 중간 |
| Repeat 딜 이전 이력 미검색 → 신규 분류 | `prevDeals` 조회 범위 제한 | 해남참농가 외 가능 | 중간 |

---

## 4. 현재 데이터 신뢰도 평가

- 월별 구매 유형 추이 차트의 **신규 수치는 과대계상 가능성** 있음 (미입력·미완료 딜 포함)
- 자금집행 완료 단계가 정확히 기록된 딜은 정상 집계됨
- "거래일" 필드 기입 관행이 팀 내 일치하지 않을 경우 오류가 추가적으로 발생할 수 있음
