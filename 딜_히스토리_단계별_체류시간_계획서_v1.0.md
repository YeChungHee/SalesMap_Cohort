# 딜 히스토리 — 단계별 체류시간 드릴다운 계획서 v1.0

작성일: 2026-05-30 KST
대상: `index.html` — 딜 성사/유예 중/지연 상세 모달(`showRealizationDetailModal`)

---

## 1. 목적

'딜 성사'(및 유예 중·지연) 모달의 **각 행(=개별 딜)을 클릭하면**, 그 딜의 **생성 → 자금집행 완료(성사)** 까지 단계별 이력을 타임라인으로 펼쳐, **각 단계에서 얼마나 지체(체류)했는지**를 보여준다.

즉 "이 딜이 어디서 오래 걸렸나"를 딜 단위로 즉시 진단한다.

---

## 2. 활용 가능한 실데이터 (이미 존재)

| 데이터 | 출처 | 내용 |
| --- | --- | --- |
| 단계 진입일 | `parseStageDwells(deal,"세일즈 파이프라인")` → `enteredTs` | 각 단계에 진입한 시각 |
| 단계 체류시간 | `parseStageDwells` → `dwellSec` | 각 단계에서 머문 누적 초(SalesMap 제공) |
| 단계 순서 | `loadPipelineMapping()` (cat: 진행→성사) | 거래 준비 → 회원가입 완료 → 신청서 작성 중 → 심사 중 → 자료 보완 중 → 실사/협의 중 → 자금집행 완료 |
| 딜 생성일 | `deal["생성 날짜"]` / `createdAt` | 딜 시작점 |
| 성사일 | 거래일 / 자금집행 완료 진입일 | 종료점 |
| 금액·업체 | `금액`, 조직 조인(orgMap) | 헤더 표시 |

추가 API 호출 불필요 — 이미 로딩된 딜 객체에서 전부 계산.

---

## 3. UX 흐름

```
딜 성사 모달 (업체 리스트)
   └─ 행(업체명·금액·성사일) 클릭
        └─ 딜 히스토리 모달 (해당 딜)
             - 헤더: 업체명 · 금액 · 생성일 → 성사일 · 총 영업사이클
             - 단계 타임라인: 각 단계 진입일시 · 체류시간 · 비중 막대
             - 가장 오래 지체한 단계 강조
             - [뒤로] 버튼으로 리스트 복귀
```

리스트의 한 행 = 한 딜이므로, 행 클릭 = 그 딜의 히스토리. (같은 업체 다건이면 각 행이 각각의 딜.)

---

## 4. 표시 항목 (딜 히스토리 모달)

### 4.1 헤더
- 업체명 · 금액
- 생성일 → 성사일
- 총 영업 사이클(일) = 생성~성사, 또는 단계 체류 합계

### 4.2 단계 타임라인 (핵심)
각 단계 한 줄:

| 단계 | 진입일시 | 체류시간 | 비중 |
| --- | --- | --- | --- |
| 거래 준비 | 26-03-16 10:20 | 2일 4시간 | ▓▓░░░ 18% |
| 신청서 작성 중 | 26-03-18 14:30 | 5일 1시간 | ▓▓▓▓▓ 41% |
| 심사 중 | … | … | … |
| 자금집행 완료(성사) | 26-04-09 | — | — |

- 체류시간 = `dwellSec`를 일/시간으로 환산
- 비중 = 해당 단계 체류 ÷ 전체 체류 합 (막대로 시각화)
- **가장 오래 지체한 단계**를 색/배지로 강조 (병목 진단)
- 진입 기록이 없는 단계는 "건너뜀"으로 표시(스킵된 단계)

### 4.3 진단 메시지(선택)
- "심사 중 단계에서 5일 지체 — 평균(P50 X일) 초과" 식으로, 기존 `STATE.cycles`(단계 P50)와 비교해 지연 단계 경고.

## 5. 데이터 연결 (구현 핵심)

현재 `wonDetails`/`pendingDetails`/`lateDetails` 항목은 `{org, won, pay, amount, state}` 만 보유 → **딜을 특정할 `dealId`가 없다.**

수정:
1. `computeRealizationModel`에서 각 상세 레코드에 **`dealId`(또는 딜 인덱스)** 를 부착.
2. `showRealizationDetailModal`의 각 행에 `onclick="showDealHistoryModal('<dealId>')"` 연결 (행 또는 업체명 클릭).
3. `showDealHistoryModal(dealId)` 신설:
   - `STATE.allDeals`에서 `id===dealId` 딜을 찾음 (`dealById` 재사용)
   - `parseStageDwells(deal,"세일즈 파이프라인")` + `loadPipelineMapping()` 단계 순서로 타임라인 구성
   - 모달 렌더 (공용 오버레이 재사용 또는 전용 오버레이)

## 6. 단계·예외 처리

- **체류초 없음**: `dwellSec`가 없으면 인접 단계 진입일 차이로 체류 추정, 그것도 없으면 "—".
- **단계 스킵**: 진입 기록 없는 중간 단계는 회색 "건너뜀"으로 표시(파이프라인을 건너뛴 딜).
- **현재 단계가 자금집행 완료 이전**(유예/지연 모달의 일부): 성사 전까지의 이력만 표시하고 마지막을 "진행 중"으로.
- **딜 못 찾음**(가져온 415건 밖): "딜 상세를 불러올 수 없음" 안내.

---

## 7. 구현 작업 분해

1. `computeRealizationModel` — 상세 레코드(`wonDetails`/`pendingDetails`/`lateDetails`)에 `dealId` 부착.
2. `showRealizationDetailModal` — 각 행을 클릭 가능하게(cursor·onclick) + 호버 표시.
3. `showDealHistoryModal(dealId)` 신설 — 단계 타임라인 렌더(진입일·체류시간·비중·병목 강조), 뒤로가기.
4. (선택) 단계 P50 대비 지연 경고 — `STATE.cycles` 활용.
5. ⓘ 설명 추가(`deal_history`) — 신규 항목 규칙대로 INFO_CONTENT 등록.

## 8. 검증

- 다운로드 딜 데이터로, 특정 딜의 단계별 `dwellSec` 합 ≈ 생성~성사 기간과 일치하는지 대조.
- 단계 진입일이 순서대로 증가하는지(역행 없는지) 확인.
- 클릭 → 히스토리 모달 → 뒤로 → 리스트 복귀 흐름 정상.
- 같은 업체 다건이 각각 다른 딜 히스토리로 열리는지.
- `node --check` 문법 검사.

## 9. 적용 범위

- 1차: **딜 성사** 모달.
- 동일 구조이므로 **유예 중·지연** 모달에도 같은 행 클릭 → 딜 히스토리를 함께 적용(성사 전 단계까지 표시).

---

## 10. 완료 기준 (DoD)

- 딜 성사 리스트의 행 클릭 시 해당 딜의 **생성→성사 단계 타임라인**이 뜬다.
- 각 단계의 **진입일시·체류시간·비중**이 표시되고 **가장 지체된 단계가 강조**된다.
- 같은 업체 다건이 각각의 딜로 정확히 구분되어 열린다.
- 유예 중·지연 모달에도 동일 적용된다.
- 추가 API 호출 없이 기존 로딩 딜로 계산된다.
