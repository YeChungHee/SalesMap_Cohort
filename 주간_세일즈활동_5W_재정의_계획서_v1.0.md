# 주간 보고서 — 세일즈 활동 5W 재정의 계획서 v1.0

작성일: 2026-05-30 KST
대상: `index.html` 주간 활동 보고 > 세일즈 활동 (`computeWeeklyActivity` / `renderWeeklyActivity`)

---

## 1. 목적

세일즈 활동 보고가 다음 5W에 **한 줄로 답**하도록 재정의한다.

| W | 의미 | 데이터 |
| --- | --- | --- |
| ① 어떤 업체 | 활동 대상 조직 | organization (조인 필요) |
| ② 언제 | 활동 발생 일시 | createdAt / date |
| ③ 무엇을 | 활동 내용(제목·요약) | memo text / todo title / email subject |
| ④ 어떤 활동 | 활동 유형 | 3대 핵심 지표로 분류 |
| ⑤ 했느냐(누가) | 수행 담당자·결과 | 담당자 + 완료/발송 상태 |

핵심 활동 지표 3가지: **DB 생성 · 고객 접촉활동 · 이메일 발송**.

> 사용자 요구 핵심: "이 내용을 확인하기 위해 **생성할 항목과 리스트**가 가장 필요함."
> → 본 계획의 중심 산출물은 (a) 5W를 채우는 **활동 행(row) 항목 정의**와 (b) 모든 활동을 한눈에 보는 **통합 세일즈 활동 리스트**다.

---

## 2. 현재 구현 진단

현재 6개 카드(DB 생성 / 상담 활동 / 심사 / TODO / 메세지 활동 / 노트 전체)는 **집계 수치**와 펼침형 "업체 상세"를 보여준다. 그러나 5W 기준으로 다음 결함이 있다.

| 결함 | 원인 | 영향 |
| --- | --- | --- |
| **① 업체명이 "—"로 빔** | `getNoteOrg`/`getActOrg`가 memo/activity 레코드의 인라인 필드(`조직`,`회사`,`organizationName`…)를 찾지만, SalesMap memo·activity API 응답에는 **조직명이 인라인으로 없음**. 텍스트 스니펫으로 폴백 → 의미 없는 값 | 누가 대상인지 알 수 없음 |
| ③ 내용이 안 보임 | 제목/요약 대신 유형 라벨 또는 스니펫만 표시 | 무엇을 했는지 모름 |
| ⑤ 담당자 없음 | 어느 카드도 수행자(영업담당)를 표시하지 않음 | 누가 했는지 모름 |
| 분류 불일치 | 6개 카드가 핵심 3지표(DB·접촉·이메일)와 1:1로 안 맞음 | 핵심 지표 판독 어려움 |
| 통합 리스트 부재 | 카드별로 흩어져 "주간 전체 활동을 한 줄씩" 볼 수 없음 | 검토·복기 비효율 |

가장 큰 문제는 ①이다. **조직 조인(entity ID → organization)** 로직이 없어, 5W의 출발점인 "어떤 업체"가 비어 있다.

## 3. 핵심 데이터 항목 정의 (활동 행 스키마)

모든 세일즈 활동을 아래 **공통 행(row)** 으로 정규화한다. 이것이 리스트의 1행 = 1활동.

| 필드 | 설명 | 소스 / 결정 규칙 |
| --- | --- | --- |
| `org_name` | ① 어떤 업체 | entity ID 조인으로 결정 (§4) |
| `org_id` | 조직 식별자 | 조인 키 |
| `occurred_at` | ② 언제 | `createdAt`/`date`/`시작일` 중 유형별 우선순위 |
| `activity_type` | ④ 어떤 활동 | 3대 분류로 매핑 (§5) |
| `sub_type` | 세부 유형 | 전화상담·고객미팅·심사·이메일·SMS·시퀀스 등 |
| `title` | ③ 무엇을 (제목/요약) | memo `text` 첫 줄, todo `제목`, email `subject` |
| `owner` | ⑤ 누가 | 담당자 (§4.2) |
| `status` | ⑤ 결과 | 완료/미완료(todo), 발송완료(email), 기록(memo) |
| `source` | 원천 | memo / todo / activity / org / lead |
| `link_id` | 원본 추적 | deal/lead/memo id (드릴다운용) |

## 4. 데이터 소스 & 조인 규칙 (핵심 수정)

### 4.1 원천 (이미 로딩됨)

| 활동 | API / STATE | 비고 |
| --- | --- | --- |
| DB 생성(조직) | `STATE.rawOrgs` | createdAt 기간 내 |
| DB 생성(리드) | `STATE.rawLeads` | createdAt 기간 내 |
| 상담·미팅·심사·노트 | `/v2/memo` (`rawWeeklyNotes`) | `유형`: 전화 상담/고객 미팅/심사 결과 |
| TODO | `/v2/todo` (`rawWeeklyTasks`) | 완료 여부·미팅 포함 |
| 이메일·SMS·시퀀스 | `/v2/deal/activity` + `/v2/lead/activity` (`rawWeeklyActs`) | `smsId`/`emailId`/`emailMessageId` |

### 4.2 조직명 조인 (① 해결 — 가장 중요)

memo/activity 레코드에는 조직명이 없고 **연결 엔티티 ID**만 있다. 다음 조인 체인으로 `org_name`을 복구한다.

1. 레코드에서 연결 키 추출: `dealId` / `deal.id` / `entityId`(entityType="deal"), `leadId` / `entityId`(entityType="lead"), 또는 `organizationId` 직접.
2. 사전(lookup) 구성 (1회):
   - `dealById` = `STATE.allDeals`(또는 rawDeals)에서 `Deal-RecordId → {organizationId, 업체명}`
   - `leadById` = `STATE.rawLeads`에서 `Lead-RecordId → organizationId`
   - `orgById`  = `STATE.orgMap` (organizationId → name) — 이미 존재
3. 해석 우선순위:
   `organizationId 직접` → `dealId → deal.organizationId → orgById` → `leadId → lead.organizationId → orgById` → `deal/lead 명칭` → `(미상)`.
4. 그래도 못 찾으면 `org_name = "(미상)"` 로 명시(스니펫 폴백 폐지).

### 4.3 담당자 조인 (⑤ 누가)

- memo/activity/todo 레코드의 `userId`/`담당자`/`createdBy` → 사용자 맵으로 이름 해석.
- 없으면 연결 딜/조직의 담당자(`Deal-담당자`, `Organization-담당직원`)로 폴백.
- (사용자 맵이 없으면 ID 그대로 표시하되 "담당자 미상" 분리.)

## 5. 활동 유형 — 3대 핵심 지표 매핑

기존 6카드를 사용자가 정의한 **3대 핵심 활동 지표**로 재그룹한다.

| 핵심 지표 | 포함 활동 | 판정 기준 |
| --- | --- | --- |
| **① DB 생성** | 신규 조직, 신규 리드 | rawOrgs/rawLeads createdAt 기간 내 |
| **② 고객 접촉활동** | 전화상담, 고객미팅, 심사, 상담 노트, (접촉형 TODO·완료) | memo `유형` ∈ {전화 상담, 고객 미팅, 심사 결과} + 접촉 메모 |
| **③ 이메일 발송** | 이메일, 시퀀스, SMS | activity `emailId`/`emailMessageId`/`smsId` 또는 type ∈ {email, sequence, sms} |

> 심사·TODO·노트 전체는 보조 지표로 유지하되, 상단 요약은 위 3지표 중심으로 표기.

## 6. 통합 세일즈 활동 리스트 (중심 산출물)

카드 하단 개별 리스트와 별도로, **주간 전체 활동을 한 행씩** 보여주는 통합 테이블을 신설한다.

### 6.1 컬럼

| 일시 | 업체 | 활동유형 | 세부 | 내용(제목/요약) | 담당자 | 상태 |
| --- | --- | --- | --- | --- | --- | --- |
| 05/27 14:20 | (주)투위 | 고객 접촉 | 전화상담 | 김민호와 간편결제 협의 | 예충희 | 기록 |
| 05/28 | 아모티 | DB 생성 | 신규조직 | — | 예충희 | 생성 |
| 05/28 09:10 | *루셀 | 이메일 발송 | 시퀀스 | 신청 안내 메일 | 자동 | 발송 |

### 6.2 기능

- 기본 정렬: **일시 내림차순**.
- 필터: 활동유형(3지표)·담당자·업체.
- 합계 배지: DB 생성 N · 고객 접촉 N · 이메일 발송 N (주간 핵심 KPI).
- 행 클릭 → 원본(노트/딜) 드릴다운(`link_id`).
- "(미상)" 업체 건수는 별도 표기해 데이터 품질 점검 신호로 활용.

### 6.3 상단 요약 카드 재편

3대 지표 카드(DB 생성 / 고객 접촉활동 / 이메일 발송)로 단순화하고, 각 카드 클릭 시 통합 리스트가 해당 유형으로 필터링되도록 한다.

## 7. 구현 작업 분해

### 7.1 `computeWeeklyActivity()` 수정

- lookup 맵 구성: `dealById`, `leadById` (orgById는 `STATE.orgMap` 재사용).
- `resolveOrg(record)` 헬퍼 신설 (§4.2) — 모든 활동에 적용.
- `resolveOwner(record)` 헬퍼 신설 (§4.3).
- 반환에 **정규화된 통합 배열 `activityRows`** 추가: `[{org_name, org_id, occurred_at, activity_type, sub_type, title, owner, status, source, link_id}]`.
- 3대 지표 카운트(`dbCreated`, `contact`, `email`)를 `activityRows` 기반으로 산출.

### 7.2 `renderWeeklyActivity()` 수정

- 상단: 3대 지표 요약 카드.
- 중앙: **통합 활동 리스트 테이블** (§6) — 정렬·필터·합계.
- `getNoteOrg`/`getActOrg`의 텍스트 스니펫 폴백 제거 → `resolveOrg` 사용.
- 행에 담당자·내용·상태 컬럼 추가.

### 7.3 데이터 품질 훅

- 조직 미해석(`(미상)`) 비율을 콘솔/배지로 노출 → 조인 키 누락 진단.

## 9. 완료 기준 (DoD)

- 모든 활동 행이 **업체·일시·내용·활동유형·담당자**를 채운다(업체 "—" 제거).
- 활동이 **DB 생성 / 고객 접촉활동 / 이메일 발송** 3지표로 분류·집계된다.
- 주간 전체 활동을 **일시 내림차순 단일 리스트**로 조회·필터할 수 있다.
- "어느 업체에, 언제, 누가, 무슨 활동을, 했는가"가 한 줄로 읽힌다.
- 조직 미해석(`(미상)`) 비율이 품질 지표로 노출된다.
