# 세일즈맵으로 노션이관 데이터 정상화 계획서

작성일: 2026-05-25

## 1. 배경

현재 세일즈맵은 2026년 2월부터 본격적으로 사용되기 시작했다. 이 과정에서 기존 Notion에 있던 고객, 리드, 딜 데이터를 세일즈맵으로 일괄 업로드했기 때문에 세일즈맵의 `생성일`이 실제 고객 유입일과 일치하지 않는 문제가 발생할 수 있다.

따라서 현재 대시보드에서 `생성일`을 기준으로 유입 코호트를 계산하면, 실제 신규 유입이 아닌 기존 이관 고객이 2026년 2월 신규 유입처럼 집계될 위험이 있다. 이 문제는 딜 전환율 100% 초과, 유입 없음 상태의 딜 존재, 딜 완료 수가 유입 수보다 크게 보이는 현상으로 이어질 수 있다.

## 2. 핵심 문제

세일즈맵 데이터의 날짜는 다음 세 가지로 분리해서 해석해야 한다.

| 구분 | 의미 | 대시보드 사용 기준 |
| --- | --- | --- |
| 실제 유입일 | 고객이 처음 유입된 날짜 | 코호트 기준월 |
| 세일즈맵 생성일 | 세일즈맵에 레코드가 생성된 날짜 | 운영/이관 로그 |
| 단계 이벤트일 | 리드 전환, 딜 성사, 자금집행 완료, 결제 완료 발생일 | 월별 성과 기준 |

현재 가장 큰 위험은 `세일즈맵 생성일 = 실제 유입일`로 간주하는 것이다. 특히 2026년 2월 일괄 업로드된 기존 Notion 데이터는 신규 유입이 아니라 `기존 이관 데이터`로 분리해야 한다.

## 3. 정상화 원칙

### 3.1 코호트 기준

코호트의 기준월은 `고객의 실제 최초 유입월`로 설정한다.

예시:

- 2025년 11월 Notion에서 최초 유입
- 2026년 2월 세일즈맵 일괄 업로드
- 2026년 5월 딜 완료

이 경우:

- 2026년 2월 신규 유입으로 집계하지 않는다.
- 코호트는 2025년 11월에 귀속한다.
- 2026년 5월 월별 성과에는 딜 완료 실적으로 집계한다.

### 3.2 이관 데이터 분리

2026년 2월에 생성된 레코드는 기본적으로 다음 중 하나로 분류한다.

| 분류 | 기준 | 처리 |
| --- | --- | --- |
| 신규 유입 | 실제 유입일이 2026년 2월 이후이고, 이관 흔적이 없음 | 코호트 포함 |
| 기존 이관 | Notion 원본 또는 메모상 과거 유입이 확인됨 | 실제 유입월로 재귀속 |
| 유입일 미확인 | 실제 유입일을 찾을 수 없음 | 별도 그룹으로 분리 |

### 3.3 월별 성과 기준

딜 성사, 자금집행 완료, 결제 완료는 코호트 기준과 별개로 `이벤트가 실제 발생한 월`에 집계한다.

예시:

- 5월 유입
- 5월 딜 성사
- 8월 딜 완료

이 경우:

- 코호트 귀속: 5월
- 코호트 표시: 5월 코호트 M+3 완료
- 월별 성과 표시: 8월 딜 완료 1건

## 4. 데이터 검증 계획

### 4.1 원천 데이터 확보

세일즈맵에서 다음 데이터를 추출한다.

- 고객/조직 데이터
- 리드 데이터
- 딜 데이터
- 단계 변경 이력
- 자금집행 완료일
- 결제 완료일
- 메모 또는 활동 로그

가능하면 기존 Notion 다운로드 원본도 함께 확보한다.

### 4.2 날짜 필드 매핑 검증

각 레코드에 대해 다음 필드를 확인한다.

| 검증 항목 | 확인 내용 |
| --- | --- |
| 세일즈맵 생성일 | 실제 유입일로 사용해도 되는지 |
| Notion 원본 유입일 | 기존 유입일 복구 가능 여부 |
| 리드 전환일 | 리드 단계 진입 시점 |
| 딜 성사일 | 자금집행 완료 또는 성사 단계 진입 시점 |
| 딜 완료일 | 결제 완료 또는 최종 완료 시점 |

### 4.3 이관 배치 탐지

다음 조건에 해당하는 데이터를 우선 이관 데이터 후보로 분류한다.

- 생성일이 2026년 2월에 집중되어 있음
- 생성일은 같지만 실제 활동, 상담, 딜 진행 이력이 과거에 존재함
- Notion 원본에 2026년 2월 이전 기록이 존재함
- 세일즈맵 생성 직후 이미 중간 단계 이상의 상태로 등록됨

### 4.4 독립 재계산

현재 구현 로직과 별도로 다음 값을 독립 계산한다.

- 실제 신규 유입 수
- 기존 이관 수
- 유입일 미확인 수
- 리드 전환 수
- 딜 성사 수
- 딜 완료 수
- 코호트별 M+0, M+1, M+2, M+3 완료율
- 월별 딜 성사/완료 실적
- 유입경로별 전환율

### 4.5 현재 화면 수치와 대조

현재 대시보드의 주요 카드와 독립 계산 결과를 비교한다.

| 화면 영역 | 검증 내용 |
| --- | --- |
| 유입 코호트 카드 | 생성일 기준으로 과대 집계되지 않았는지 |
| 리드 전환 | 실제 리드 단계 진입 기준인지 |
| 딜 성사 | 자금집행 완료 기준과 혼합되지 않았는지 |
| 딜 완료 | 결제 완료 기준인지 |
| 코호트 전환율 | 분모와 분자가 같은 기준인지 |
| 유입경로별 표 | 딜 수가 고객 수 분모를 초과하지 않는지 |
| 월별 구매 유형 | 이벤트 발생월 기준인지 |

## 5. 이상치 점검 기준

다음 항목은 별도 이상치 리포트로 추출한다.

- 딜 전환율이 100%를 초과하는 유입경로
- 유입 수는 0인데 딜 또는 완료가 존재하는 고객
- 세일즈맵 생성일은 2026년 2월이지만 과거 활동 이력이 있는 고객
- 실제 유입일이 없는 고객
- 딜 완료일이 딜 성사일보다 빠른 고객
- 리드 전환 없이 딜 성사 또는 딜 완료가 존재하는 고객
- 동일 고객 또는 동일 딜의 중복 업로드 가능성
- 결제 완료 수가 코호트 고객 수보다 큰 경우

## 6. 대시보드 반영 방향

### 6.1 카드 구조

대시보드 카드는 다음 기준으로 분리한다.

| 카드 | 기준 |
| --- | --- |
| 신규 유입 코호트 | 실제 최초 유입월 |
| 기존 이관 고객 | Notion에서 이관된 기존 고객 |
| 유입일 미확인 | 실제 유입일이 없는 고객 |
| 월별 딜 성사 | 딜 성사 이벤트 발생월 |
| 월별 딜 완료 | 딜 완료 이벤트 발생월 |
| 평균 영업 사이클 | 실제 유입일 또는 리드 전환일부터 완료일까지 |

### 6.2 필터 구조

기간 필터는 최소 두 가지 모드를 제공하는 것이 바람직하다.

| 필터 모드 | 설명 |
| --- | --- |
| 코호트 기준 | 실제 유입월 기준 |
| 성과 발생월 기준 | 딜 성사/완료 이벤트 발생월 기준 |

이 두 기준을 혼합하면 전환율 해석이 왜곡되므로, 화면에서 명확히 분리 표시해야 한다.

## 7. 구현 수정 방향

### 7.1 데이터 모델

각 고객 또는 조직 레코드에 다음 정규화 필드를 추가한다.

| 필드 | 설명 |
| --- | --- |
| `actual_inflow_date` | 실제 최초 유입일 |
| `salesmap_created_at` | 세일즈맵 생성일 |
| `migration_flag` | Notion 이관 여부 |
| `migration_source` | 원본 출처 |
| `inflow_confidence` | 실제 유입일 신뢰도 |
| `cohort_month` | 코호트 귀속월 |
| `event_month_deal_won` | 딜 성사 발생월 |
| `event_month_deal_completed` | 딜 완료 발생월 |

### 7.2 유입일 결정 우선순위

실제 유입일은 다음 우선순위로 결정한다.

1. Notion 원본의 최초 유입일
2. 세일즈맵의 명시적 유입일 또는 상담 시작일
3. 최초 활동/메모/상담 기록일
4. 최초 리드 생성일
5. 세일즈맵 생성일

단, 5번은 이관 데이터가 아닌 신규 생성 데이터일 때만 사용한다.

### 7.3 이관 데이터 처리

이관 데이터는 다음 규칙으로 처리한다.

- 실제 유입일이 복구되면 해당 월 코호트로 재귀속한다.
- 실제 유입일이 없으면 `유입일 미확인`으로 분리한다.
- 이관 데이터는 2026년 2월 신규 유입 수에 포함하지 않는다.
- 월별 성사/완료 실적에는 실제 이벤트 발생월 기준으로 포함한다.

## 8. 검증 산출물

검증 후 다음 산출물을 작성한다.

1. 세일즈맵 원천 데이터 필드 매핑표
2. Notion 이관 데이터 식별표
3. 유입일 정상화 전/후 비교표
4. 현재 대시보드 수치와 독립 계산 수치 비교표
5. 이상치 목록
6. 수정 필요 로직 목록
7. 대시보드 카드 기준 정의서

## 9. 최종 판단 기준

현재 데이터 유효성 검증의 핵심 판단 기준은 다음과 같다.

```text
2026년 2월 세일즈맵 생성일은 실제 유입일로 간주하지 않는다.
기존 Notion 일괄 업로드 데이터는 기존 이관 데이터로 분리한다.
코호트 기준은 실제 최초 유입일로 계산한다.
월별 성과 기준은 각 단계 이벤트 발생일로 계산한다.
유입일이 불명확한 데이터는 코호트 계산에서 별도 분리한다.
```

이 기준으로 재계산했을 때 현재 대시보드 수치와 차이가 발생하면, 해당 차이는 단순 오류가 아니라 `기준일 정의 오류`, `이관 데이터 미분리`, `분모/분자 기준 불일치` 중 하나로 분류해 수정해야 한다.
